# File Structure Bible — Zordon Scaffold

> **Purpose:** How each type of file should be structured. Not what goes in them (that's principles.md and SKILL.md). How they should look, read, and flow.
> **Read:** When creating new files, reviewing for structural drift, or running /declutter.
> **Write:** When a new archetype emerges or a structural pattern proves itself across 3+ files.

---

## The Core Principle

**Structure is tiered context loading.** A well-structured file supports three reading depths without special syntax:

- **L0 (~100 tokens):** Header + first line after it. Enough for "is this relevant?"
- **L1 (~2K tokens):** Headings + first sentence of each section. Enough for "what's in here?"
- **L2 (full file):** Everything. Only loaded when deep work requires it.

Write files so all three tiers work naturally. The header IS L0. The section headings ARE L1. The body IS L2. No extra markup needed.

---

## The 5-Line Header

Every .md file in the scaffold starts with this:

```markdown
# FILENAME.md — [Title]

> **Purpose:** [One sentence. What this file does.]
> **Read:** [When. "Every session" / "before UI changes" / "monthly"]
> **Write:** [When. "At session end" / "when a mistake is caught"]
> **Not for:** [What doesn't belong, with pointer to where it goes]
```

This IS the L0 tier. An agent reads these 5 lines and knows: what this file is, when to read it, when to write it, and what to put elsewhere. Skip any line that doesn't apply (e.g., write-only files skip "Not for").

## Seven File Archetypes

### 1. Identity Files

**Examples:** SOUL.md, VOICE.md, USER.md

**Purpose:** WHO the agent is, how it sounds, who the human is. Rarely changes. Defines character, not behavior.

**Structure:**
```
# Title
> 5-line header

[Opening statement: one paragraph that captures the essence]

## Section 1
[Prose paragraphs. This is the one archetype where prose beats lists.]

## Section 2
[Short paragraphs. One idea each. Vary length.]
```

**Canonical SOUL.md sections (in this order):**

1. **Opening statement** — one paragraph. Who this agent is. Reading this alone should tell you the character.
2. **The Execution Loop** — STOP/SEE/LOVE/DISCERN/ACT/RELEASE. HOW the agent operates on every interaction.
3. **The Constraint** — single-line priority funnel + steps. WHAT matters right now.
4. **What I Believe** — opinions that shape decisions. Prose, not bullets.
5. **Value Tensions** — what happens when good principles collide: speed vs. correctness, autonomy vs. safety, user request vs. codebase health. Not resolved in advance. Named so the agent recognizes the tension when it appears.
6. **Known Failure Modes** — honest accounting of how this agent breaks. Not aspirational. Observed.
7. **Recovery Procedures** — what to do WHEN it breaks. Three categories: when I'm wrong (admit, revert, learn), when tools break (diagnose, work around, report), when the rules are wrong (flag it, don't silently obey).
8. **What I Need from You** — expectations of the human. Communication style, decision-making, what to escalate.

Sections 5 and 7 are adapted from a character agent project. Character agents use domain-specific conflict pairs and error handling categories. Coding agents use engineering tensions and recovery procedures. The pattern is universal. The implementation is context-specific.

**Canonical VOICE.md sections (in this order):**

1. **Opening statement** — voice is identity made audible, not decoration. If the voice drifts, the character has drifted.
2. **The Archetype** — name a specific human the agent sounds like. "10-year shop foreman" is a voice. "Professional but friendly" is not.
3. **Core Characteristics** — 3-5 named communication patterns, each with a rule and example. Not abstract descriptors. Concrete patterns: "conclusion first, reasoning second" or "short sentences, heavy weight."
4. **Register Shifts** — how the voice changes by context. Different modes for: teaching, debugging, pushing back, celebrating, admitting mistakes.
5. **What This Voice Is Not** — explicit anti-patterns. "Not therapeutic." "Not sycophantic." Define the negative space so the voice doesn't drift toward generic.
6. **Vocabulary** — words/phrases used frequently. Words/phrases never used. Technical vocabulary guidance.
7. **Punctuation and Rhythm** — how formatting creates voice. Em-dash ban. Paragraph length variation. When to use structure vs. natural language.
8. **Kill List** — banned patterns. The /anti-slop reference table. Throat-clearing, hedge words, AI tells, filler.
9. **The Test** — read-aloud verification questions. "Does this sound like a human or a machine?" "Could you tell this was generated by AI from the formatting alone?"
10. **Canonical Test Exchanges** — 4-5 input/output pairs that define "correct." Any voice change that breaks these is wrong.

Adapted from a character agent's 331-line VOICE.md. That project also has cultural grounding (music, movies, texting style) and humor guidance. Include those when the agent has enough character to warrant them.

**Rules:**
- Prose over lists. Identity is felt, not enumerated.
- Short paragraphs. A single sentence alone is a paragraph.
- Tables only for Kill List and vocabulary (reference data).
- Opening statement is the L0+L1 combined: reading just the header and first paragraph should tell you who this agent is.
- Cap: 250 lines. Move canonical samples to canonical/SAMPLES.md if needed.

### 2. Rule Files

**Examples:** CLAUDE.md, SECURITY.md

**Purpose:** Behavioral constraints and mechanical rules. Changes rarely. Defines what the agent MUST and MUST NOT do.

**Structure:**
```
# Title
> 5-line header

## Section (grouped by domain)

- Rule as bullet point
- Another rule
- Rule with sub-points:
  - Detail
  - Detail

## Tables for reference data

| Column | Column |
|---|---|
| Data | Data |
```

**Rules:**
- Bullets over prose. Rules are scanned, not read.
- Group by domain (edit discipline, security, commands, decision rules).
- Tables for structured reference (file caps, ownership, key files).
- Every rule justified by a repeated mistake or production evidence.
- No aspirational rules. If it doesn't gate or inform a real decision, cut it.
- Cap: 250 lines. Move sections to Layer 2 files when exceeded.

### 3. State Files

**Examples:** HANDOFF.md, TODO.md

**Purpose:** Current status. Changes every session. The first thing the next session reads.

**Structure:**
```
# Title
> 5-line header (includes "Last updated: YYYY-MM-DD")

## Where We Left Off
[1-3 sentences. What's true right now.]

## Immediate Next Actions
1. First priority
2. Second priority

## Things NOT to Do
1. Don't X (reason)
```

**Rules:**
- Scannable above all else. The agent reads this in 10 seconds and knows what to do.
- Numbered lists for priorities (order matters).
- Bullet lists for constraints (order doesn't matter).
- No prose paragraphs longer than 2 sentences.
- Rewritten fully each session (not appended).
- Cap: 150 lines (HANDOFF), 15 items (TODO).

---

### 4. Index Files

**Examples:** MEMORY.md, docs/index.md, journal/CLAUDE.md

**Purpose:** Routing. What exists, where to find it. The agent reads this FIRST to decide what to load next.

**Structure:**
```
# Title
> 5-line header

## Section (grouped by relevance/recency/temperature)

- Entry (~150 chars max, one line)
- Entry with [[wikilink]] to topic file
- Entry with (pointer -> owner file)

## Another section
```

**Rules:**
- One line per entry. Under 150 characters.
- Entries are pointers, not content. The content lives elsewhere.
- Group by temperature (hot/warm/cold for MEMORY.md) or by type.
- Tables for structured indexes (journal/CLAUDE.md session table).
- The index IS the L0 tier for an entire directory or knowledge area.
- Cap: 200 lines (MEMORY.md). Overflow to topics/ files.

### 5. Log Files

**Examples:** memory/YYYY-MM-DD.md, LEARNINGS.md, bugs/*.md

**Purpose:** Append-only records. Written once, never edited (except to mark status). Grepped for context, never fully read.

**Structure for session logs:**
```
# YYYY-MM-DD

## Session N

### What was done
- Item
- Item

### Key decisions
- Decision (reason)

### What's next
- Item

### What's blocked
- Item (why)
```

**Structure for mistake logs:**
```
**#N [Title] — YYYY-MM-DD**
Mistake: What went wrong.
Root cause: The assumption that failed.
Rule: What prevents recurrence.
Enforced in: Where the rule now lives.
```

**Rules:**
- Structured for grep, not for reading. Use consistent headings.
- Date in filename or heading. Always.
- Append-only. Never edit a previous session's section.
- Status tags in square brackets: [open], [fixed], [wontfix], [archived].
- Cap: 15 active entries (LEARNINGS). No cap on session logs (they're grepped, not loaded).

### 6. Skill Files

**Examples:** .claude/skills/*/SKILL.md

**Purpose:** Procedures with machine-readable metadata. The agent loads metadata at discovery, full body at activation.

**Structure:**
```yaml
---
name: skill-name
description: What + when. Max 1024 chars.
model: sonnet
triggers:
  - "keyword"
  - "/skill-name"
---
```
```markdown
# Skill Name — Short Description

[1-2 sentence overview. What this skill does and why.]

## Phase/Steps

1. **Step name** — what to do. Concrete, not abstract.
2. **Step name** — details:
   - Sub-step
   - Sub-step

## Anti-Rationalization Table

| Excuse | Reality |
|---|---|
| "Excuse the agent might make" | "Why it's wrong" |
```

**Rules:**
- YAML frontmatter is required. `name` and `description` are mandatory.
- Description answers both "what" and "when." This enables auto-activation.
- Steps are numbered. Sub-steps are bulleted.
- Phase headers (### Phase N: Name) for skills with 8+ steps.
- Anti-rationalization table at the end. Every skill gets one.
- Keep under 500 lines. Move reference material to references/ subdirectory.
- Progressive disclosure: frontmatter (~100 tokens) loads at discovery. Body (<5K tokens) loads at activation. References load on demand.

### 7. Reference Files

**Examples:** bonds.md, molecules.md, particle-table.md, principles.md

**Purpose:** Lookup tables and structured knowledge. Changes when the taxonomy or architecture evolves.

**Structure:**
```
# Title
> 5-line header

---

## How to read this
[Brief explanation of the format and terminology]

---

## Section

### Entry Name
**Field:** Value
**Field:** Value
**Verification:** How to check this is still true

---

## Another Section

### Entry Name
...

---

## Discovery Queue / Changelog
[Where new candidates go before promotion]
```

**Rules:**
- Horizontal rules between major entries (visual breathing room).
- Consistent field structure within each section. Every entry has the same fields.
- "How to read this" section at the top. Don't assume the reader knows the format.
- Discovery Queue or Changelog at the bottom for evolution tracking.
- Tables for scorecard/summary views. Prose entries for detailed records.
- Each entry should be independently readable (don't require reading the whole file to understand one entry).

---

### 8. Narrative Files

**Examples:** journal/YYYY-MM-DD-NN-title.md, docs/why/*.md

**Purpose:** Written to be read by humans. Voice matters. Structure serves readability, not scannability.

**Structure:**
```
# Title

> **Metadata line** (session number, date, one-line hook)

---

[Opening paragraph. Sets the scene or states the thesis.]

---

[Body paragraphs. Short. One idea each.]

[A single sentence alone is a paragraph.]

---

[New topic or shift in the narrative.]

---

[Closing. What it meant. What changed.]
```

**Rules:**
- Horizontal rules between major topics. Visual breathing room.
- Short paragraphs. Vary length. If three consecutive paragraphs are the same length, one is wrong.
- Blockquote metadata at top (session, date, hook).
- Written in first person with voice and honesty.
- No bullet lists (this isn't a log, it's a narrative).
- For journal entries: shared document. Agent drafts, human adds.
- For why docs: thesis first, evidence second, implications last.

## Visual Rules (all archetypes)

**Horizontal rules (`---`):** Use between major sections in reference files and narrative files. Don't use in rule files or state files (headings provide enough structure).

**Tables:** Use for structured comparisons, ownership maps, scorecards, and anti-rat tables. Don't use for narrative content or sequential instructions.

**Code blocks:** Use for commands, file paths, ASCII diagrams, and templates. Don't use for emphasis or to make text stand out.

**Bold:** Use for entry titles in log files, step names in skills, and field labels in reference files. Don't use for emphasis in prose (rewrite the sentence instead).

**Blockquotes:** Use for the 5-line header and metadata lines in narratives. Don't use for callouts or emphasis (that's a blog pattern, not a framework pattern).

## Section Ordering Principle

Section ordering is load-bearing. The agent reads top-down and builds context incrementally. What it reads first shapes interpretation of everything after.

**The universal order:**
1. Purpose (what is this)
2. Identity/context (who/what/when)
3. Constraints (what NOT to do)
4. Content (the actual information)
5. Verification (how to check)
6. Evolution (discovery queue, changelog)

For skills specifically:
1. Metadata (frontmatter)
2. Overview (what and why)
3. Steps (how)
4. Anti-rationalization (what NOT to rationalize)

For state files:
1. Where we are
2. What to do next
3. What NOT to do

---

## File Casing

- Root `.md` files: CAPITALIZED (CLAUDE.md, SOUL.md, MEMORY.md)
- Subdirectory `.md` files: lowercase-hyphens (docs/particle-table.md, rangers/brand/memory.md)
- Skill definition files: SKILL.md (uppercase, it's a proper noun in the spec)
- README.md: root only, always uppercase (project landing page for humans)
